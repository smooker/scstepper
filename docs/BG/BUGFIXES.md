# Поправени грешки — scstepper фърмуер

Завършените елементи са показани със ~~зачертаване~~ и дата на поправка.
Отворените елементи са без зачертаване.

## ~~БУГ-01: Decel рампата никога не стартира (decelIndex=0)~~ — поправен 2026-03-13

**Симптом:** Моторът ускорява нормално, но никога не забавя — работи при максимална
скорост до последната стъпка, след което спира рязко.

**Причина:** `decelIndex` се инициализираше на 0 (като `accelIndex`), така фазата на
забавяне незабавно удовлетворяваше условието "готово" и се пропускаше изцяло.

**Поправка:** Инициализиране на `decelIndex` с правилното отместване в рамп таблицата
за фазата на забавяне.

## ~~БУГ-02: R1 — Flooding на промпта при спиране на jog/step мотор~~ — поправен 2026-03-20

**Симптом:** След CDC команда `move`/`mover`/`movel`, пускането на jog бутон (или всяко
последващо спиране на мотора) отпечатваше допълнителен промпт, заливайки терминала.

**Причина:** `PrintPrompt()` се викаше безусловно при всяко спиране на мотора, независимо
дали е активна CDC команда.

**Поправка:** Добавен флаг `cdcMoveActive` — промптът се отпечатва само при завършване
на CDC-инициирано движение, не при jog/step спирания.

## ~~БУГ-03: Зумерът бипва при всяко натискане/пускане на бутон~~ — поправен 2026-03-20

**Симптом:** Зумерът се задействаше при всяко EXTI събитие на jog/step бутон (натискане
и пускане), което правеше невъзможно разграничаването на обратна връзка от бутон от
удари в краен изключвател.

**Причина:** `buzzRequest = 1` се задаваше безусловно в началото на EXTI ISR, преди
switch/case по пин.

**Поправка:** Преместен `buzzRequest = 1` само в case-овете за ES_L и ES_R — зумерът
сега бипва изключително при удари в краен изключвател.

## ~~БУГ-04: JOG_STEP брой импулси недетерминистичен (41–79 вместо 80)~~ — поправен 2026-03-20

**Симптом:** Saleae capture (`capture_smooker.sr`, 100 kHz) показа 43 burst-а;
само 6 имаха точно 80 импулса. Останалите варираха от 41 до 75.

**Причина:** `Stepper_Stop()` се викаше безусловно в EXTI UP (пускане) ISR handler-а за
jog бутони — дори по средата на активно JOG_STEP движение. Пускането на бутона преди
завършване на 80-импулсното движение го прекъсваше.

**Поправка:** Добавена проверка `jogStateL/R == JOG_CONT` в UP ISR handler-а:
```c
if (snapA & BUTT_JOGL_Pin) {
    if (jogStateL == JOG_CONT) Stepper_Stop();  /* спира само непрекъснатия jog */
    lastTick_jogL = now;
    evtFlags |= EVT_JOGL_UP;
}
```
След поправката: 43/43 burst-а = точно 80 импулса. ✓

**Верифицирано чрез:** `captures/capture_smooker.sr` (сравнение преди/след)

## ~~БУГ-05: JOG_CONT race condition — моторът се движи до краен изключвател при пускане~~ — поправен 2026-03-20

**Симптом:** Пускането на jog бутон по време на JOG_CONT понякога нямаше ефект —
моторът продължаваше да работи до удар в краен изключвател.

**Причина:** Race между ISR и main loop:
1. UP ISR се задейства докато `jogStateL == JOG_PRESSED` → без `Stepper_Stop()` (поради guard-а от БУГ-04)
2. Main loop след това преминава `jogStateL → JOG_CONT` и вика `Stepper_RunContinuous()`
3. Бутонът вече е пуснат — повече UP събитие не пристига → моторът работи безконтролно

**Поправка:** Добавена предпазна мрежа в UP handler-а на `ProcessEvents()`:
```c
if (flags & EVT_JOGL_UP) {
    if (jogStateL == JOG_CONT) Stepper_Stop();  /* предпазна мрежа */
    jogStateL = JOG_IDLE;
}
```
Ако main loop изпревари, предпазната мрежа го хваща при обработка на UP събитието.

## ~~БУГ-06: Грешна честота на HSE кристал в CubeMX — USB clock 50 MHz вместо 48 MHz~~ — поправен 2026-03-20

**Симптом:** Произволни/изкривени символи след USB re-enumeration (reconnect на кабел
или USB reset от host). Фърмуерът работеше отначало, но се корумпираше след всяко прекъсване.

**Причина:** `.ioc` имаше `RCC.HSE_VALUE=24000000`, но действителният кристал на
Black Pill платката е **25 MHz**. При PLLM=12:

| Настройка | Грешно (24 MHz) | Правилно (25 MHz) |
|-----------|----------------|------------------|
| VCO_in | 2 MHz | 1 MHz |
| VCO_out | 192 MHz | 192 MHz |
| SYSCLK | 96 MHz | 96 MHz |
| USB (÷4) | **50 MHz ✗** | **48 MHz ✓** |

USB получаваше 50 MHz вместо изискваните 48 MHz, причинявайки грешки в bit-timing-а,
проявяващи се като произволни символи след re-enumeration.

**Поправка:** `scstepper.ioc`: `PLLM=25`, `PLLN=192` → VCO_in=1 MHz, USB=48 MHz точно.

## ~~БУГ-07: Несъвместимост на FPU директивата в стартовия асемблер~~ — поправен 2026-03-20

**Симптом:** Потенциална ABI несъвместимост при стартиране — декларацията на FPU в
асемблера е несъвместима с флаговете на компилатора. Може да причини hard fault ако FPU
регистрите се използват преди инициализацията на C runtime.

**Причина:** `startup_stm32f411xe.s` имаше `.fpu softvfp`, докато `Makefile` компилираше с
`-mfpu=fpv4-sp-d16 -mfloat-abi=hard`.

**Поправка:** Сменено на `.fpu fpv4-sp-d16` в стартовия асемблер за съответствие с флаговете на компилатора.

## ~~БУГ-08: NVIC приоритети зададени runtime вместо в CubeMX~~ — поправен 2026-03-20

**Симптом:** Функцията `FixNVIC_Priorities()` се викаше в `main()` след `MX_GPIO_Init()`
за корекция на неправилно генерирани от CubeMX приоритети. Крехко решение — всяка
регенерация от CubeMX би върнала тихо грешните приоритети.

**Причина:** NVIC приоритетите не бяха конфигурирани в `scstepper.ioc` — CubeMX
default-ите бяха грешни и се коригираха ръчно в C код.

**Правилни приоритети (PRIORITYGROUP_4, по-малко число = по-висок приоритет):**

| IRQ | Приоритет | Причина |
|-----|-----------|---------|
| TIM2 | 0 | Stepper pulse ISR — най-висок |
| ES_L (EXTI3), ES_R (EXTI4) | 0 | Трябва да спре мотора атомарно с TIM2 |
| JOG_L (EXTI9_5) | 0 | Същото |
| STEP_L (EXTI0), STEP_R (EXTI1) | 2 | По-нисък — без timing-критично спиране |
| USB OTG_FS | 3 | Най-нисък от групата |

**Поправка:** Добавени правилни NVIC записи в `scstepper.ioc`; `FixNVIC_Priorities()` премахната изцяло.

## ~~БУГ-09: USB PCD callbacks — BKPT, липсващо USBD_LL извикване, безполезен SOF~~ — поправен 2026-03-20

**Под-проблем A — BKPT в suspend callback:**
`My_PCD_SuspendCallback` съдържаше инструкция `BKPT` (debug артефакт). Въпреки
че не беше регистриран в продукция, dead code с breakpoint trap е опасен. Премахнат.

**Под-проблем B — Disconnect callback без USBD_LL_DevDisconnected:**
`My_PCD_DisconnectCallback` задаваше `CDC_IsConnected = 0` но не викаше
`USBD_LL_DevDisconnected()` — USB device стекът никога не биваше уведомяван за
disconnect, оставяйки вътрешното му състояние корумпирано при следваща enumeration.
Поправка: добавено `USBD_LL_DevDisconnected((USBD_HandleTypeDef *)hpcd->pData)`.

**Под-проблем C — SOF callback беше no-op:**
`My_PCD_SOF` беше регистриран като callback, но пропускаше `USBD_LL_SOF()`.
Регистрацията премахната; default handler в `usbd_conf.c` вика `USBD_LL_SOF` правилно.

**Под-проблем D — Излишни DataOut и Connect callbacks:**
`MyPCD_DataOutStageCallback` и `My_PCD_ConnectCallback` дублираха default HAL
поведение без да добавят нищо. Премахнати.

## ~~БУГ-11: CubeMX usbd_conf.c — излишни предупреждения -Wunused-parameter~~ — поправен 2026-03-20

**Симптом:** При всяка компилация се появяваха предупреждения:
```
usbd_conf.c: unused parameter 'size' in USBD_static_malloc
usbd_conf.c: unused parameter 'p' in USBD_static_free
```

**Причина:** CubeMX-генерираните `USBD_static_malloc` / `USBD_static_free` използват
статичен буфер — параметрите `size` и `p` са умишлено игнорирани. В тези функции
няма USER CODE секции, така че `#pragma GCC diagnostic` не може да се добави без да
се изтрие при следваща регенерация.

**Поправка:** Per-file правило в `Makefile` с `-Wno-unused-parameter` само за `usbd_conf.o`.
Оцелява при CubeMX регенерация (Makefile не се пипа от CubeMX).

## БУГ-R2 (отворен): командата `range` не изключва бутоните

**Симптом:** Физическите бутони остават активни по време на измерването `range`.
Натискане на jog бутон по време на `RunRange()` причинява недефинирано поведение
(прекъснато измерване).

**Необходима поправка:** Задаване на `buttonsEn = 0` в началото на `RunRange()`,
възстановяване при завършване.

## БУГ-R3 (отворен): `moveto` приема команда докато моторът работи

**Симптом:** Изпращане на `moveto <mm>` докато вече е активно движение поставя второ
движение без interlock — резултатът е недефиниран (позицията се корумпира, потенциален удар в ES).

**Необходима поправка:** Проверка на `Stepper_IsBusy()` при влизане в handler-а на `moveto`;
отпечатване на грешка и връщане ако моторът не е idle.
