|------|--------|-----------------|--------|

Accel/Decel съотношение: **2.02x** (очаквано 2.0x — перфектно!)

### Асиметрия — анализ

Забавянето е 2× по-стръмно от ускорението (dvdtdecc/dvdtacc = 100/50 = 2).
Измерено:

```
Трапец (5 mm):     accel = 153.9 ms, decel = 76.3 ms  → ratio = 2.02x  ✓
Триъгълен (1 mm):  accel = 123.8 ms, decel = 53.7 ms  → ratio = 2.30x  (~2x + гранични ефекти)
```

При кратките движения съотношението е малко по-високо (2.3x вместо 2.0x) заради ефектите от дискретизацията на рамп таблицата при ниски скорости.

### Визуален профил (ASCII)

```
Скорост (mm/s)
  10 |         ___________________________
     |        /                           \
   8 |       / accel=50 mm/s²    decel=100 \  mm/s²
     |      /                               \
   4 |     /                                 \
     |    /                                   \
   1 |___/                                     \___
     |
     └──────────────────────────────────────────── Време
      0    154 ms        518 ms         594 ms
           ←accel→       ←const→       ←decel→
           325 ст.       1512 ст.       162 ст.
```

Забавянето заема **половината** от времето на ускорението, но използва **половината** от стъпките — двойно по-стръмен наклон на скоростната крива.

## 3. Квантоване и грешки

### Честота на дискретизация

| Честота | Резолюция | Грешка при 250 µs | Грешка при 2500 µs |
|---------|-----------|--------------------|--------------------|
| 100 kHz | 10 µs | ±4.0% | ±0.4% |
| 1 MHz | 1 µs | ±0.4% | ±0.04% |

Захващанията при 1 MHz (capture_combo_final_speed.sr) дават значително по-точни резултати.

### Систематични грешки

1. **Измереният min период е 240 µs вместо 250 µs** — фазово подравняване между TIM2 (96 MHz) и FX2 (48 MHz, от USB SOF). Разлика = 1 семпъл при 100 kHz, 10 семпла при 1 MHz → вътрешно закръгляне на TIM2 ARR.

2. **Максимален период 2148 µs вместо 2500 µs** — първият и последният импулс от рампата може да не попаднат в захващането (setup delay).

## 4. Заключения

1. **Броят импулси е точен** — spmm=400, всяко движение дава точния брой стъпки
2. **Асиметричните рампи работят коректно** — dvdtdecc=2×dvdtacc дава 2× по-стръмно забавяне
3. **Accel/Decel ratio = 2.02x** за трапецовиден профил (перфектно съвпадение с теорията)
4. **Триъгълният профил** коректно не достига mmpsmax при кратки движения
5. **Двете посоки са идентични** — DIR инверсия не влияе на timing-а

## 5. Jog бутони — bounce анализ (2026-03-19)

### Открит бъг: bounce повторно натискане при пускане

При отпускане на jog бутон механичното отскачане генерира фалшив LOW фронт след реалния HIGH. Тъй като `lastTick_jogL/R` се обновявал само при натискане (не при пускане), bounce повторното натискане минавало дебаунс проверката след всяко задържане > 30ms → второ нежелано движение.

```
BUTT_JOGR_Pin:
                  ┌─────────────────────┐   ┌──┐
                  │       ЗАДЪРЖАНЕ     │   │bounce│
──────────────────┘                     └───┘  └─────────
                  ▲ натискане           ▲ пускане  ▲ bounce натискане
                  lastTick=now          │          │
                                        │          now - lastTick >> DEBOUNCE_MS
                                        │          → EVT_JOGR_DN → второ движение ✗
```

### Поправка (2026-03-19)

Нулиране на `lastTick` при пускане — bounce натискането вижда `< DEBOUNCE_REL_MS`:

```
                  ┌─────────────────────┐   ┌──┐
                  │       ЗАДЪРЖАНЕ     │   │bounce│
──────────────────┘                     └───┘  └─────────
                  ▲ натискане           ▲ пускане  ▲ bounce натискане
                  lastTick=now          lastTick=now   │
                                                       now - lastTick < DEBOUNCE_REL_MS
                                                       → филтрирано ✓
```

Засяга: кратък jog (JOG_STEP) и дълъг hold (JOG_CONT). И двата бутона (L и R).
